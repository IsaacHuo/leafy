# 安全与隐私

MyLeafy 涉及学校身份、课表与学业信息、社区内容和管理权限。安全设计的核心不是“隐藏入口”，而是明确数据权威、最小化上传范围，并让权限在可信边界中成立。

## 数据边界

| 数据 | 默认处理方式 |
|---|---|
| 学校密码 | 仅用于学校登录流程，不上传到 MyLeafy 业务后端 |
| 教务 Cookie / 会话 | 保留在设备受控存储与系统 Cookie 环境中 |
| 课表、成绩、考试 | 本地缓存；除用户主动共享的必要字段外默认不上云 |
| 社区资料与内容 | 存储在 Supabase，受 RLS 与校园范围控制 |
| 图片 | 私有 Storage bucket，通过受控路径和 signed URL 访问 |
| AI API Key | 保存于设备 Keychain，不交给 Tool Gateway |
| 管理会话 | HttpOnly、Secure、SameSite Cookie；不进入浏览器 storage |

## 信任边界

### 学校系统

学校系统是教务数据的权威来源，但页面和登录链路并非稳定 API。解析错误不能被静默当作“没有数据”。

### iOS 客户端

客户端持有用户授权会话和本地数据。日志、错误弹窗、诊断包和分享内容不得泄露 Cookie、密码、完整 token、原始 HTML 或不必要的个人信息。

### Supabase

客户端 publishable key 不是秘密。真正的授权来自 RLS、资源所有权、校园范围与 Edge Function 校验。

### 运营后台

浏览器不直接获得服务端密钥。管理请求经过同源代理与 Edge Functions，并按角色、资源、校园和动作记录审计。

## AI 与公开检索

- 模型回答是辅助信息，不代表学校官方结论。
- 只有用户允许的本机上下文才能进入模型请求。
- 公开搜索词和网页内容可能经过工具网关，但模型 API Key 与本机校园数据不得发送给该网关。
- 对网页、PDF 或外部数据的读取应明确来源、限制和失败状态，不能声称分析了未实际读取的正文。
- AI 动作必须收敛，并在不可逆操作前由用户确认。

## 开发检查清单

- [ ] 客户端和浏览器只使用 publishable key。
- [ ] 暴露表启用并测试 RLS，包含拒绝路径。
- [ ] 多校园数据验证 `campus_id`。
- [ ] Storage 使用私有 bucket 和 signed URL。
- [ ] 更新操作不能修改 owner/profile/campus 字段逃逸授权。
- [ ] 日志不记录密码、Cookie、完整 token 或真实学生数据。
- [ ] 管理写操作有参数白名单、最小角色和审计记录。
- [ ] 导出限制字段与行数，并防止表格公式注入。
- [ ] Issue、PR 和测试夹具使用脱敏或合成数据。

发现安全问题时，不要在公开 Issue 粘贴密钥或真实用户数据，请使用仓库列出的维护者联系方式进行最小化披露。
