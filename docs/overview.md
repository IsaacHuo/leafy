# MyLeafy 项目总览

本文给出 MyLeafy 的产品定位、系统组成、当前能力与责任边界。实现细节见[架构说明](architecture.md)，页面和交互规则见[App 产品设计](app-design.md)。

## 1. 产品定位

MyLeafy 是以高校课表和学业数据为核心的原生 iOS 应用，目前主要面向北京林业大学。它解决的不是单一“查课表”问题，而是将分散在教务网页、个人记录、校园信息和同学交流中的高频任务整合为连续工作流：

1. 通过学校教务身份进入应用。
2. 获取并缓存用户主动授权访问的课表与学业数据。
3. 在课表、学业、社区和个人空间中完成日常操作。
4. 由 Leafy AI 在用户允许的本机上下文上提供解释、整理与计划辅助。
5. 使用 Supabase 同步需要跨设备或多人协作的社区、通知、共享和评价数据。

产品对外名称为 **MyLeafy**；仓库名、Xcode target 与部分历史类型仍使用 `leafy` / `Leafy`。

## 2. 产品原则

### 2.1 课表是核心上下文

课程时间、地点、周次和学期是大量校园任务的共同基础。课表不仅是展示页面，也为提醒、分享、Widget、学习记录、AI 上下文和共享课表提供结构化输入。

### 2.2 学校数据与 MyLeafy 数据分开

学校教务系统仍是课表、成绩、考试和培养数据的权威来源。Supabase 只保存 MyLeafy 自己产生或用户明确发布的数据，不替代学校教务，也不将本地课表缓存自动视为公共数据。

### 2.3 本地优先，远程能力按需启用

课表、成绩缓存、提醒与部分个人记录可在本地继续读取。社区、通知、共享和运营能力依赖 Supabase。远程能力不可用时，应用应缩小功能范围，而不是阻断所有本地体验。

### 2.4 原生交互优先

页面以 SwiftUI 系统导航、列表、Tab、sheet、alert 和 SF Symbols 为基础。新系统视觉能力只作为增强，并通过可用性检查保留 iOS 17 基线行为。

### 2.5 能力按校园配置

项目已引入校园描述与能力开关。北京林业大学拥有完整的教务适配和主要社区能力；自定义校园或其他校园只显示其明确支持的功能，避免展示无法兑现的入口。

## 3. 系统组成

| 子系统 | 主要职责 | 权威数据 |
|---|---|---|
| iOS App | 登录、课表、学业工具、社区、AI、个人设置与离线缓存 | 用户设备上的本地数据与界面状态 |
| 学校教务系统 | 身份验证、课表、成绩、考试、教学计划等 | 学校教务数据 |
| Supabase | 社区身份、内容、通知、共享、评价、配置与后台业务 | MyLeafy 业务数据 |
| 运营后台 | 内容治理、目录维护、配置、审计与受控导出 | 对 Supabase 业务数据的管理视图 |
| 官网与分享页 | 产品介绍、公开链接和有限的分享落地页 | 可公开展示的数据 |

## 4. 当前能力范围

### 4.1 身份与校园

- 北京林业大学强智教务登录、验证码与会话恢复。
- 学校会话和 Supabase 会话分离管理。
- 匿名 Supabase 用户与教务学号对应的社区 profile 绑定。
- 社区邮箱绑定与验证；邮箱是 MyLeafy 业务联系方式，不替代学校密码验证。
- 校园能力描述、自定义校园导入和按校园隐藏不支持的入口。

### 4.2 课表与时间

- 当前周课表、周次切换、课程详情和重叠课程布局。
- 学期、周、月等时间范围视图及考试、倒计时、提醒等日程叠加。
- 课程备注、提醒设置、课表背景和单日/周期分享。
- Widget 快照与深链跳转。
- 共享课表快照、一次性邀请码、只读授权和撤销关系。
- 远程学期运行配置，本地保留最近成功配置与内置兜底。

### 4.3 学业与个人学习

学业根入口按主题组织工具，而不是把所有页面平铺在根 Tab 中。当前代码包含以下主要领域：

- 教学培养：成绩、成绩分析、考试、荣誉记录、综合测评、教学计划和培养方案。
- 时间日程：日程报告、校历、考试与自定义倒计时。
- 空闲教室：空教室查询、指定教室占用、校园热力相关能力和学习时间记录。
- 学习空间：学习项目、资料、任务和记录。
- 体育相关：阳光长跑、体测记录和体育场馆。
- 职业与考研：职业规划内容和考研公开信息。
- 评价相关：教师、课程及其他按校园开放的结构化评价。
- 校园扩展：医疗事项、周末出行等特定校园工具。

具体入口根据校园、后端能力和当前版本配置动态显示。

### 4.4 Leafy AI

- 原生对话界面和本机对话记录。
- 在用户允许范围内读取本机学业上下文。
- 支持自备 API Key；密钥保存于 Keychain。
- 当前请求由设备直接发送到 DeepSeek；订阅、托管额度和联网搜索暂未开放。
- 可生成报告、清单、表格、流程说明等 Artifact。
- Artifact 在独立阅读页渲染 Markdown、图表或公式，并支持受支持格式的导出。
- AI 动作保持收敛：允许导航、准备提醒或生成建议，不直接篡改学校原始数据和社区内容。

AI 输出是辅助信息，不是学校官方结论；成绩、考试、培养要求等最终以学校系统为准。

### 4.5 社区与通知

- 帖子流、文本与图片发布、详情、评论、点赞、投票和收藏。
- 全局置顶、分类置顶、软删除和内容状态控制。
- 评论、点赞等站内通知与公告。
- 个人资料、我的帖子、评论、点赞、投票等内容视图。
- 分享链接与 App 深链。
- 意见反馈和必要的社区访问限制。

### 4.6 运营与管理

- React-admin 管理界面和按资源加载的管理页面。
- `viewer`、`operator`、`super_admin` 角色。
- 社区内容、用户、公告、反馈、目录数据、运行配置和管理员会话管理。
- 服务端筛选的 CSV 导出、请求追踪和审计日志。
- Cloudflare Pages Functions 代理与 HttpOnly 管理会话。

## 5. 数据来源与所有权

| 数据 | 来源 | 默认存储 | 是否会同步到 Supabase |
|---|---|---|---|
| 学校身份与会话 | 学校教务登录 | Keychain/系统 Cookie 与本机状态 | 仅同步业务绑定所需的标识，不同步学校密码 |
| 课表、成绩、考试、培养数据 | 学校教务系统 | SwiftData 或本地缓存 | 默认不上传；共享课表仅上传用户主动发布的字段子集 |
| 课程备注、提醒、学习记录 | 用户设备 | 本地持久化 | 仅特定功能明确启用同步时上传 |
| 社区资料和内容 | 用户在 MyLeafy 中创建 | Supabase | 是 |
| 图片 | 用户主动上传 | 私有 Storage bucket | 是，通过受控路径和 signed URL 访问 |
| 通知、公告、反馈 | MyLeafy 业务 | Supabase | 是 |
| 教师与课程目录 | 运营导入或审核 | Supabase | 是 |
| AI Key 与本机对话 | 用户设备 | Keychain / 本地存储 | 自备 Key 与历史记录不上传；当前不启用托管调用 |
| 管理审计 | 运营后台 | Supabase | 是，仅授权管理员可读 |

## 6. 信任边界

### 学校教务

MyLeafy 是学校网页的客户端适配层，不控制教务服务的可用性、页面结构和认证策略。学校返回内容是教务事实来源，但解析器需要面对非稳定 HTML。

### iOS 客户端

客户端持有用户授权的教务会话和本地缓存，因此必须避免日志、错误弹窗或分享内容泄露 Cookie、学号、原始 HTML 和其他敏感数据。

### Supabase

iOS 使用 publishable key 和用户会话。授权依赖 RLS、profile 归属、校园范围和 Edge Function 校验，不能把客户端 UI 隐藏视为安全控制。

### 运营后台

后台是高权限系统。浏览器不直接持有服务端密钥；所有高权限操作必须经过 Cloudflare 代理和 Edge Functions，并写入审计记录。

## 7. 当前限制与风险

| 风险 | 影响 | 当前策略 |
|---|---|---|
| 教务 HTML 或登录流程变化 | 登录、课表或学业解析失败 | 多入口识别、解析回归样本、可诊断错误与课表 WebView 兜底 |
| 校园网络或学校服务不可达 | 无法刷新教务数据 | 保留本地缓存，区分网络、会话和解析错误 |
| 客户端发起学号绑定 | 改包场景下身份强度有限 | 社区业务使用 RLS；更高风险能力应迁移到服务端验证 |
| 目录数据依赖人工维护 | 教师/课程搜索不完整 | 模板导入、后台审核和数据来源记录 |
| 功能范围快速增长 | 导航复杂、文档漂移 | 校园能力配置、领域分组、契约测试与文档同步要求 |
| 管理权限误用 | 内容或用户数据受影响 | 角色最小权限、确认交互、服务端白名单和审计 |

## 8. 项目状态与发展方向

MyLeafy 已形成完整的 iOS、Supabase 和 Web 管理三端架构，当前工作重点是稳定性和边界治理，而不是继续增加根级入口：

- 提高教务解析的兼容性、测试覆盖与故障恢复质量。
- 统一不同校园的数据适配协议和能力配置。
- 完善学习空间、课表与日程之间的数据连接。
- 收紧身份绑定、共享、后台导出和管理操作的安全边界。
- 减少重复状态与重复文档，使实现、测试和文档保持一致。

以上是方向而非版本承诺，优先级会随真实使用反馈调整。
