# MyLeafy 未来功能展望

本文从用户价值出发，记录 MyLeafy 可能继续发展的产品能力。它描述值得验证的问题、最小可用范围和实施边界，不代表版本承诺或固定发布日期。

当前实现以[项目总览](overview.md)和[App 功能总结](Leafy_APP功能总结.md)为准；稳定性、安全和工程治理优先级见[发展路线图](roadmap.md)。

## 1. 核心判断

MyLeafy 已经覆盖课表、学业数据、学习空间、社区、共享课表、校园工具和 Leafy AI。下一阶段最重要的工作不是继续堆叠入口，而是让已有数据和功能形成连续工作流：

- 用户打开 App 后能立刻知道今天发生什么、下一步要做什么。
- 一门课程的课表、资料、任务、考试、提醒和学习记录能够互相连接。
- Leafy AI 能理解用户明确授权的上下文，并把建议转化为可预览、可确认的操作。
- 学校或网络暂时不可用时，最近成功的数据仍然可信、可读、可恢复。
- 新能力优先进入现有“课表”“校园”“Leafy”页面，不新增根 Tab。

## 2. 优先级概览

| 优先级 | 方向 | 主要价值 | 进入实施的前提 |
|---|---|---|---|
| P0 | 今日与本周工作台 | 降低每天确认课程、考试、任务和提醒的成本 | 复用现有课表与日程数据，不新增根入口 |
| P0 | 课程空间闭环 | 把课程、资料、任务、专注记录和 AI 串在一起 | 统一课程标识和本地数据关系 |
| P0 | 数据刷新与恢复中心 | 让用户知道数据是否新鲜、失败在哪里、如何恢复 | 错误分类、最近成功时间和缓存状态可靠 |
| P0 | Leafy AI 可确认动作 | 让回答真正进入学习流程，同时保留用户控制 | 所有写操作先预览、后确认、可撤销 |
| P1 | 选择性跨设备同步 | 降低换机和多设备使用成本 | 明确同步范围、冲突策略与删除语义 |
| P1 | 更轻量的协作能力 | 支持同学间共享空闲时间、课程安排和学习计划 | 最小字段、限时授权、随时撤销 |
| P1 | 系统级入口 | 通过 Widget、快捷指令和系统搜索更快到达高频功能 | 不复制主 App 的复杂状态 |
| P1 | 校园信息订阅 | 减少反复查找官网通知和关键日期的成本 | 可靠来源、更新时间和失效处理 |
| P2 | 多校园适配框架 | 让其他学校复用通用能力 | 校园适配协议、维护责任和隔离测试成熟 |
| P2 | 隐私可控的个性化 | 在本机或最小化数据范围内提供更合适的提醒与建议 | 用户授权、可解释来源和关闭入口完整 |

P0、P1、P2 表示建议顺序，不对应具体版本。

## 3. 近期：先把已有能力连成闭环

### 3.1 今日与本周工作台

在课表页现有结构中增加可折叠的“今天 / 本周”摘要，统一展示：

- 下一节课、地点变化和课程备注。
- 即将到来的考试、截止事项和自定义日程。
- 今日学习任务、最近专注记录和未完成提醒。
- 数据最后更新时间，以及需要重新登录或刷新的状态。

最小版本只聚合现有数据并提供跳转，不引入新的任务系统。用户进入 App 后应在一个屏幕内完成“确认安排—发现冲突—进入处理”。

### 3.2 课程空间

课程详情不再只承担信息展示，可逐步成为一门课程的工作入口：

- 关联本地资料、任务、课程备注和专注记录。
- 展示考试、成绩与教学计划中的对应信息，并标明数据来源。
- 从课程上下文进入 Leafy AI，用于解释知识点、整理复习清单或生成阶段总结。
- 允许用户自行建立无法自动匹配的课程关系，并支持随时解除。

第一阶段应只做本地关联和统一入口。自动匹配必须提供确认，不能因为课程名称相近就直接合并数据。

### 3.3 数据刷新与恢复中心

MyLeafy 依赖非稳定的学校网页，刷新是否成功本身就是重要产品状态。建议统一展示：

- 课表、成绩、考试、培养方案等数据的最近成功更新时间。
- 当前使用的是实时结果、本地缓存还是内置参考数据。
- 网络不可达、学校会话过期、页面变化和解析失败等不同原因。
- 单项重试、重新认证和保留缓存的明确后果。

该能力可放在“我的—同步与安全”中，并在具体页面保留简短状态入口。目标是减少“数据看起来正常，但已经过期”的风险。

### 3.4 Leafy AI 可确认动作

Leafy AI 已能结合本机校园上下文和生成 Artifact，下一步可以强化“建议到行动”的闭环：

- 从回答中准备学习任务、自定义日程、提醒或课程备注草稿。
- 执行前展示字段、影响范围和数据存储位置。
- 用户逐项确认后再写入，成功后提供撤销或跳转查看。
- 对成绩、学校课表、社区发布和账号权限继续保持只读或强确认边界。

AI 的价值应来自减少整理和录入成本，而不是替用户作出不可逆决定。

## 4. 中期：增强连续性与协作

### 4.1 选择性跨设备同步

当前大量个人数据只保存在本机。可优先评估同步用户自己创建且恢复价值高的数据：

- 课程备注、提醒、自定义日程和学习任务。
- 学习项目的结构化信息，不默认同步原始资料文件。
- 个性化设置和已确认的课程关联。

学校密码、Cookie、完整教务 HTML、自备模型 Key 和未经选择的教务数据不进入同步范围。同步设计必须先定义冲突合并、退出账号、删除设备和彻底删除数据的语义。

### 4.2 更轻量的同学协作

在现有共享课表基础上，优先发展低风险、目标明确的协作：

- 计算双方共同空闲时间，只返回时间段，不暴露完整课表。
- 为小组学习创建限时共享日程或任务清单。
- 对共享内容提供到期时间、查看范围和撤销入口。
- 默认不展示课程备注、成绩、学习记录等私人字段。

暂不建议建设私信或群聊。即时通信会带来持续在线、审核、举报、未成年人保护和通知治理成本，与项目当前核心优势不匹配。

### 4.3 系统级入口

围绕高频、短操作补充 Apple 平台能力：

- 锁屏或桌面 Widget 展示下一节课、考试倒计时和刷新状态。
- App Intents / 快捷指令用于打开今日课表、开始专注记录、查询指定教室。
- Spotlight 或系统搜索索引用户明确允许暴露的本地课程与日程。
- 仅在确有持续实时价值时评估 Live Activity，避免为了视觉效果常驻后台。

系统入口只承载摘要和跳转，复杂编辑仍回到主 App。

### 4.4 校园信息订阅

把分散的学校通知转化为可追溯的信息订阅：

- 用户按学院、教务、竞赛、就业、奖助学金等主题订阅。
- 每条内容展示来源、发布日期、抓取时间和原文链接。
- 对失效页面、重复内容和过期通知进行明确处理。
- AI 可以摘要和解释，但原文始终是事实依据。

没有稳定来源或维护责任的频道不应上线；不得把搜索结果直接包装成官方通知。

## 5. 长期：为扩展能力建立基础

### 5.1 多校园适配框架

多校园不等于复制页面后替换学校名称。成熟的适配框架至少需要：

- 统一的校园描述、学期、课表、成绩、考试和教室协议。
- 校园能力清单、数据来源声明和运行时配置。
- 每个适配器独立的解析样本、契约测试和故障开关。
- Supabase 数据、社区内容与运营权限严格按校园隔离。
- 明确的适配维护者和退出机制。

在上述条件不足时，应只开放导入课表和通用本地工具，不承诺完整教务接入。

### 5.2 隐私可控的个性化

个性化可以用于提醒强度、工作台排序和学习建议，但需要遵循：

- 尽量在设备端计算，服务端只接收完成业务所需的最小字段。
- 告诉用户建议使用了哪些数据，并允许关闭某类上下文。
- 不根据成绩、作息或社区行为生成带有羞辱性、歧视性或过度确定的标签。
- 推荐结果可被忽略、纠正和重置，不形成不可见的永久画像。

### 5.3 数据质量与运营能力

随着校园和公共数据增多，需要把数据质量当作独立能力：

- 为目录和通知记录来源、更新时间、审核状态和适用校园。
- 对重复、冲突和过期数据提供后台治理流程。
- 建立关键配置变更的审计、回滚和影响预览。
- 使用匿名化的失败率、恢复率和功能使用信号判断优先级，不收集无关个人数据。

## 6. 暂不建议投入的方向

- 新增更多根 Tab，继续把低频工具平铺在一级导航。
- 建设完整即时通信、校园外卖、二手交易或支付系统。
- 将学校密码、Cookie 或完整教务数据集中上传到 MyLeafy 后端。
- 依赖不可追溯的聚合内容或不稳定第三方页面作为权威数据。
- 让 AI 自动修改学校数据、自动发帖、代替用户报名或执行高风险操作。
- 只因技术新颖就引入常驻后台、复杂推荐模型或重型服务端架构。

这些方向并非永远排除，但它们当前的维护、安全与合规成本明显高于对核心校园工作流的价值。

## 7. 功能进入开发前的判断标准

每个候选功能至少回答以下问题：

1. 它解决的是高频问题，还是偶尔出现的需求？
2. 能否复用现有课表、校园、Leafy 或我的页面？
3. 数据来自哪里，谁是权威来源，多久会过期？
4. 离线、学校会话失效和服务端不可用时如何表现？
5. 是否新增个人数据、共享关系或高权限操作？
6. 最小版本能否在较短周期内验证用户价值？
7. 谁负责长期维护适配、数据和文档？
8. 上线后用什么指标判断保留、调整或下线？

建议优先观察：

- 用户从打开 App 到确认今日安排所需的操作数。
- 课表、成绩和考试刷新成功率，以及失败后的恢复率。
- 课程资料、任务和提醒被重复使用的比例。
- AI 草稿被确认、修改、撤销的比例，而不是只看调用次数。
- 新功能是否降低入口分散和重复录入，而不是单纯增加页面数量。
