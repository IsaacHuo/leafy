# Leafy 后续路线图

本文档记录当前实现之后的优先工作。当前事实见 [项目总览](overview.md) 和 [架构说明](architecture.md)。

## 1. 当前阶段目标

Leafy 已经从单纯 UI 验证推进到“教务直连 + Supabase 社区 + 后台运营”的组合形态。下一阶段重点不应继续扩散新入口，而应稳定核心链路：

- 课表和教务解析稳定性。
- 社区、通知、反馈、评教的线上可运营性。
- TestFlight 前的隐私、错误、配置和数据安全检查。
- UI 细节 polish，特别是错误态和空态。

## 2. 优先级

### P0：TestFlight 前必须处理

- 确认 Release/TestFlight 不会暴露原始 Cookie、HTML 调试内容或敏感配置。
- 检查 `Config/Leafy.local.xcconfig` 只存在本地，不提交真实 Supabase key。
- 验证强智登录、验证码、课表、成绩、考试、教学培养、自习室都能在目标网络下工作。
- 验证 Supabase migration、Edge Functions、匿名登录、Email provider、Storage bucket 均已部署到同一个项目。
- 完成 [TestFlight 检查清单](testflight-checklist.md)。

### P1：核心体验稳定

- 继续加强课表抓取失败时的用户文案，区分学校网络、登录态、解析变化。
- 对 `HTMLParser` 增加更多真实页面样本回归测试。
- 优化首次登录后的数据同步反馈。
- 让课表空状态、无课状态和解析失败状态更清楚。
- 继续打磨今日课程摘要、分享图和自习室联动。

### P2：社区和评教运营

- 后台已具备内容下架、恢复、用户禁言、反馈处理、公告发布、帖子置顶和客户端 Feed 预览；后续重点是按真实运营反馈优化筛选和批量处理。
- 导入并维护教师名录，建立最小可用的教师数据更新流程。
- 评教继续保持星级评分边界；暂不引入文字评价和复杂审核流。
- 通知中心继续完善公告和社区通知的已读、忽略、静音行为。

### P3：产品扩展

- 帖子收藏已接入 Supabase；自习室收藏继续保留为本地工具能力。
- 缓存与同步页提供更清晰的手动刷新入口。
- 数据安全页补充学校数据、社区数据、本地缓存和退出登录后的处理说明。
- 根据真实使用反馈再决定是否新增功能，不新增根 Tab。

## 3. 不做或暂缓

短期不进入：

- 用 Supabase Auth 替代教务登录。
- 把学校登录迁入后端。
- 做评教文字评价、匿名评论、复杂排序和审核推荐系统。
- 引入独立消息推送基础设施。
- 引入 0class 后端作为 Leafy 业务后端。

需要更强安全边界时再评估：

- 后端代管教务登录校验。
- 教务学号绑定防改包伪造。
- 服务端抓取教师名录或教务静态资料。

## 4. 验收标准

下一阶段完成时应满足：

- 用户在可达学校网络下能稳定登录并看到课表。
- 教务功能失败时，用户知道该重新登录、切换网络还是稍后重试。
- 社区 profile、帖子、图片、评论、点赞、通知和评教能在 TestFlight 环境真实操作。
- 后台能完成最基本的社区内容运营和公告发布。
- 文档能准确指导新开发者理解当前架构和部署边界。
