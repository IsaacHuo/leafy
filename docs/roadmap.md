# MyLeafy 发展方向

本文记录项目可能继续推进的方向，不代表固定版本、日期或交付承诺。当前实现与限制以[项目总览](overview.md)和[架构说明](architecture.md)为准。

## 近期重点

### 教务稳定性

- 建立更多脱敏 HTML 样本和解析回归测试。
- 进一步区分网络、会话、中间页和 DOM 变化。
- 优化已有缓存下的刷新、恢复与错误说明。
- 减少对单一页面路径和脆弱选择器的依赖。

### 学业工作流

- 统一课表、考试、倒计时、学习项目和记录的时间语义。
- 减少功能重复入口，让学业领域保持清晰。
- 为成绩、学分和统计结果补全口径、来源和时间范围。
- 继续优化课表投影和复杂页面的渲染性能。

### 多校园基础

- 把校园差异收敛到 capability、描述和适配器。
- 提供更明确的自定义校园导入格式和验证反馈。
- 保证 Supabase 数据、Feed 和管理权限始终按校园隔离。
- 在增加新校园前先定义数据来源、维护责任和降级行为。

### 安全与隐私

- 评估将高价值学校身份验证迁移到可信服务端。
- 收紧共享、导出、邮箱绑定和管理 action 的最小权限。
- 完善敏感日志审查、数据保留和账号退出语义。
- 为关键 RLS 和管理拒绝路径增加自动化测试。

### 文档与工程质量

- 保持 D2 架构图、代码边界和文档一致。
- 减少历史兼容 facade 和重复状态模型。
- 扩大协议注入和纯 Domain 测试覆盖。
- 让 Web、Edge Functions、数据库和 iOS 的契约变化可被自动检查。

## 可能的后续方向

- 更成熟的课表与日程协作能力。
- 更可靠的离线编辑与跨设备同步策略。
- 基于用户明确授权的个性化学习辅助。
- 面向更多校园的通用适配框架。
- 对社区、目录和评价数据质量的持续治理。

是否实施由真实需求、数据来源、安全成本和维护能力共同决定。

## 暂不作为默认方向

- 用 Supabase Auth 替代学校教务登录。
- 把学校原始密码或完整教务数据集中上传到 MyLeafy 后端。
- 为追求功能数量继续增加根 Tab。
- 开放匿名文字评价而不建立审核和法律风险控制。
- 让 AI 自动执行不可逆操作或绕过用户确认发布内容。
- 在没有可靠数据来源和维护责任时开放校园专属入口。

## 判断标准

新增方向进入实施前，应回答：

1. 它解决的高频用户问题是什么？
2. 数据来自哪里，谁是权威来源？
3. 失败、离线和会话失效时如何降级？
4. 是否需要新的个人数据或管理权限？
5. 能否归入现有信息架构而不增加根级复杂度？
6. 谁负责长期维护数据、适配和文档？
